1 # Overview
2 This repository contains the code for the MotorController board, developped
3 by Telecom Robotics. This board is part of the [robot-framework](https://github.com/TelecomParistoc/robot-framework) currently developped by the association. It can also be used independantly from this framework.
4 This board, and the associated code, are in charge of handling all low-level tasks
5 related to robot moves: motor control, data acquisition from various sensors
6 (IMU and coding wheels for now), current position computation and so on.
7 
8 Below, you'll find the general documentation for the various components of this project. More detailed documentation is available [here](https://telecomparistoc.github.io/MotorController/)

10 # Download
11 In order to get a working copy of this code, you have to follow some steps in
12 addition to simply clone this repo. Indeed, some generic drivers (the IMU driver
13 for now) are included as a submodule.
14 The interest of submodules is that they allow you to always get the most up-to-date
15 version of a project developped in an independant repository. And keeping repository
16 focused on a single project appears to us as a good practice: one repository for one project.
17 
18 Run the following commands in a terminal to get a fully usable copy of this project:
19 - clone the repository
20 - cd in the directory of the repo
21 - ```$ git submodule init```
22 - ```$ git submodule update```
23 
24 At any time when using this repo, if you want to obtain the latest updates of the generic
25 drivers, you can use ```$ git submodule update --remote```.

27 # Compilation & flash
28 The same code is used for both robots. Data which are specific to each robot (PID
29 coeffs, motor configuration, ...) are stored in flash and loaded when the program
30 starts (see _Configuration_ below for more details on that point).
31 
32 __Note:__ You're going to compile on your computer a program that will run on another
33 processor, with a different architecture and different capabilities. Such a process
34 is called __cross-compilation__. To do it, you need a "special" version of gcc, one
35 which is specifically designed to run on a standard Linux/Mac machine but to
36 generate binaries for a different architecture (an ARM in our case).
37 The version we use is called _arm-none-eabi-gcc_ (_gdb_, _ld_, _as_ ... also have their
38 version for cross-compilation, with the same prefix. You're probably wondering what these cryptic terms mean. Let's take them one after the other:
39  - _arm_ : because our target architecture (the one on which the sofware built will
40  run) is ARM.
41  - _none_ : because we don't use a "high-level" OS (their are also linux version for
42  instance).
43  - _eabi_ : this indicates which standard is used for functions calls, parameters passing, registers usage, etc...). It stands for _Embedded-Application Binary Interface_.
44 
45 Let's go to practice now:
46 
47 Simply run ```$ make``` to compile the code (it's equivalent to ```$ make all```).
48 List of *make* targets available:
49  - **all**: compile the code
50  - **clean**: delete all files produced by the firmware compilation process
51  - **interface**: compile the interface generator and run it
52  - **interface_clean**: delete all files produced by the generator compilation
53  - **distclean**: delete all files that can be generated, useful to restore a
54  clean environment
55  - **debugserver**: launch JLinkGDBServer and waits for a GDB connection
56  - **gdb**: start gdb with the binary file of the project
57 
58 To flash a board, you need:
59  - a JTAG probe (any JTAG probe should work, but this documentation assumes that
60  you use a SEGGER one).
61  - a MotorController board to flash.
62  - a 12V power source for the board.
63 
64 The JTAG probe is used to communicate with the microprocessor embedded in the board.
65 It allows you to flash a new firmware and to debug it (with gdb for instance).
66 
67 Then, follow these steps:
68  - connect the SEGGER probe to the board with the SWD cable and to your computer
69  with the USB cable.
70  - power up the board
71  - go the parent directory of this repository (the one containing the Makefile)
72  - run ```$ make``` in a terminal
73  - run ```$ make debugserver```
74  - open a new terminal
75  - run ```$ make gdb``` in this new terminal
76 
77 In addition, to see the printf output, you need JLinkRTTClient. You should already
78 have it as it usually comes with JLinkGDBServer).
79 In another terminal, run ```$ JLinkRTTClient```.
80 
81 Note: If you don't have JLinkGDBServer, you can download it [here](https://www.segger.com/downloads/jlink-beta/)
82 (choose *J-Link Software and Documentation Pack*).

91 
92 Note: Welcome to the embedded world, you can forget powerful user-friendly OS like Linux.
93 ChibiOS is small and doesn't provide as many tools as a desktop OS. One of the most
94 surprising things when you start working on a deeply embedded project such as
95 MotorController is that the OS is fully integrated in the binary of your program.
96 The firmware and the OS are not separated, as it is the case in a desktop computer
97 for instance. So each time you flash a new version of your software, you also flash
98 the OS.

100 ## Communication
101 This component is in charge of the communication between the central unit of the
102 robot (a Raspberry Pi in the actual implementation) and the motorboard.
103 It acts as an I2C slave, providing an interface that looks like memory-mapped registers.
104 For the descriptions of this interface, see the file [specs.md](./specs.md).
105 
106 It's composed of 2 files:

111 ## Coding wheels
112 This component is responsible of handling the coding wheels inputs. It's basically
113 an interrupt handler.
114 It's composed of 2 files:
115  - *coding_wheels.h*
116  - *coding_wheels.c*
117 
118 To initialize this driver, call the ```init_coding_wheels()``` function. As
119 parameter, pass a `coding_wheels_config_t` object. This object must contain the
120 following information:
121  - initial value of each counter (right wheel and left wheel)
122  - orientation of each wheel (DIRECT or INDIRECT). This parameter allows the user
123  to decide which rotation sense corresponds to a forward movement and thus to
124  an increase of the associated counter).
125 
126 **Note**: It's necessary to enable the ChibiOS EXT driver by setting `HAL_USE_EXT` to
127 `TRUE` in *halconf.h*
128 
129 This driver then displays two counters (one for each coding wheel): `right_ticks`
130 & `left_ticks`.

152 ## Control
153 The goal of this component is to manage the low-level part of the robot movements.
154 It will receive orders from the robot central unit and control the motors to
155 execute them.
156 
157 It's composed of 2 files:
158  - *control.h*
159  - *control.c*
160 
161 This module defines 2 threads that run in parallel. One is in charge of computing
162 intermediate orders based on the "high level" orders sent by the master and on the
163 ettings defined (max acceleration, cruise speed...). The other is responsible of
164 applying a PID on the motors command based on the current order and on data read
165 from sensors (coding wheels and IMU for the moment).

169 ## Configuration
170 Some configuration variables are used in order to provide the requested services.
171 These data are stored in flash (non-volatile memory), so that they can be kept across
172 reset. When booting up, the firmware loads the values from the flash and initialise
173 the configuration variables with them.
174 The I2C interface provides several registers to update these values.
175 If you want to keep the newly updated values for subsequent execution, it's necessary
176 to store them in flash. To do so, simply set the desired values to the variables
177 and then, write any value to the **STORE_DATA_IN_FLASH** register (see the
178 description of the communication interface in [specs.md](./specs.md) to get more info on this
179 register).
